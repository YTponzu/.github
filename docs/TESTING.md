# テスト戦略ガイド（共通）

## 方針

- **スタイル：TDD**（テストを先に書いてから実装）
- **カバレッジ方針：バランス型**（ビジネスロジック・APIはしっかり、UIは最低限）
- **E2E：プロジェクトの規模・要件に応じて判断**

---

## テストの種類と優先度

| 種類 | 対象 | 優先度 | ツール |
|------|------|--------|--------|
| ユニットテスト | ビジネスロジック・ユーティリティ関数 | ⭐⭐⭐ 高 | Vitest |
| 統合テスト | API・DBアクセス・hooks | ⭐⭐⭐ 高 | Vitest |
| コンポーネントテスト | 重要なUIの動作・インタラクション | ⭐⭐ 中 | Vitest + React Testing Library |
| E2Eテスト | クリティカルなユーザーフロー | ⭐ 低（必要な場合のみ） | Playwright |

---

## TDDサイクル

```
1. Red   — 失敗するテストを書く
2. Green — テストが通る最小限の実装をする
3. Refactor — コードを整理する（テストは通ったまま）
```

### 実践のコツ

- テストは「何をするか」を日本語で書いてから実装する
- 一度に大きく作らない。小さいサイクルを繰り返す
- リファクタリングはテストが通っている状態でのみ行う

---

## フレームワーク選定基準

### Next.js プロジェクト

| ツール | 用途 |
|--------|------|
| **Vitest** | ユニット・統合・コンポーネントテスト |
| **React Testing Library** | コンポーネントのインタラクションテスト |
| **Playwright** | E2E（導入する場合） |

### Node.js バックエンド

| ツール | 用途 |
|--------|------|
| **Vitest** | ユニット・統合テスト |
| **Supertest** | HTTPエンドポイントのテスト |

---

## カバレッジ目標

| 対象 | 目標 |
|------|------|
| ビジネスロジック・サービス層 | 80%以上 |
| APIエンドポイント・DB操作 | 80%以上 |
| UIコンポーネント | 重要なインタラクションのみ |
| ユーティリティ関数 | 90%以上 |

カバレッジの数値より「重要なロジックが守られているか」を優先する。

---

## テストファイルの配置

各機能・モジュールと同じディレクトリに `__tests__/` または `.test.ts` を置く。

```
src/features/agents/
├── hooks/
│   ├── useAgents.ts
│   └── useAgents.test.ts     ← hooks のテスト
└── components/
    ├── AgentCard.tsx
    └── AgentCard.test.tsx    ← コンポーネントのテスト
```

---

## E2E導入判断基準

以下のいずれかに当てはまったら Playwright を導入する：

- [ ] ユーザーが直接触れるクリティカルなフロー（決済・認証等）が増えた
- [ ] バグが本番環境で繰り返し発生するようになった
- [ ] チームが増えて手動確認のコストが高くなった
